# 掘金文章 —— 前端 / 可视化

**标题：** 我把玄奘西行17年5万里做成交互故事：前端如何呈现历史可视化

**封面建议：** story3.html 地图飞越截图（金色路线从长安延伸到印度，带叙事卡片）

---

## 缘起：真实唐僧的故事，比西游记震撼十倍

做丝绸之路知识图谱的第三周，我决定做一个新功能：交互故事。

传统的可视化是上帝视角——你俯视整张地图，所有数据一目了然。但历史不是这样被体验的。历史是一段一段走出来的，是一站一站到达的。

我选了三个丝路上最重要的人物：张骞、郑和、玄奘。

第三个故事——玄奘西行，是我写得最投入的。

如果你只知道《西游记》里的唐僧，你可能不知道真实的玄奘有多猛：

- 唐朝初年严禁出境，他是**偷渡**出去的
- 穿越莫贺延碛八百里流沙沙漠，水袋打翻，**四天五夜滴水未进**，差点死掉
- 高昌国王哭着求他留下当国师，他**绝食三日**拒绝，国王最终放行并资助
- 翻越帕米尔高原雪山，同行者冻死过半
- 在印度那烂陀寺学习五年，成为**全印度佛学辩论第一名**
- 17年后带回 **657部梵文佛经**，唐太宗亲自出城迎接

一个人，没有孙悟空，没有白龙马。5万里，110个国家。

**我觉得这个故事值得被更多人看到——用前端技术，而不只是文字。**

## 技术方案

### 整体架构

还是那句话：**单一数据源 → 多种呈现形态**。玄奘的路线数据已经在 JSON 里了：

```json
{
  "id": "xuanzang-journey",
  "name": "玄奘西行",
  "year": 629,
  "dynasty": "唐",
  "route": ["changan", "dunhuang", "gaochang", "kucha", "kashgar",
            "samarkand", "taxila", "nalanda", "khotan"]
}
```

交互故事本质上是对这组数据的**逐章叙事包装**——每一章锁定一个城市节点，地图飞到那个节点，弹出故事卡片。

### 核心实现：地图叙事引擎

```
┌─────────────────────────────────────┐
│  Leaflet 地图层                      │
│  ┌───────────────────────────────┐  │
│  │  路线层 (Canvas)               │  │
│  │  节点标记 (CircleMarker)      │  │
│  │  飞越动画 (flyTo)             │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │  叙事卡片层 (DOM absolute)     │  │
│  │  - 章节标题                    │  │
│  │  - 故事正文                    │  │
│  │  - 关联实体标签                │  │
│  │  - 上一章/下一章 导航          │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

关键交互逻辑：

```javascript
function goToChapter(n) {
  const chapter = chapters[n];
  // 1. 地图飞越到目标城市
  map.flyTo([chapter.lat, chapter.lng], chapter.zoom, {
    duration: 2.5,  // 2.5秒飞越，不赶不拖
    easeLinearity: 0.25
  });
  // 2. 等飞越完成后弹出叙事卡片
  map.once('moveend', () => {
    showStoryCard(chapter);
    // 3. 累计绘制已走路线（不中断不空白）
    drawRouteProgress(chapters.slice(0, n + 1));
  });
}
```

### 三个小细节，体验差别巨大

**1. 路线累计绘制**

这是最容易踩坑的地方。如果每章单独画一段路线，用户会看到路线**断断续续**，完全丧失了"一个人走了多远"的累积感。

正确做法：**路线是累加的**。到第3章时，第1-3章的路线都在；到第7章时，整条路线完整呈现。

```javascript
// ❌ 错误：每章画一段
drawRoute([chapter.start, chapter.end]);

// ✅ 正确：累计画所有已走过的
drawRoute(allChapters.slice(0, currentIndex + 1).map(c => c.route));
```

**2. 飞越时长与叙事节奏的匹配**

`flyTo` 太快（0.5s）像闪现，太慢（5s）用户着急。实测 **2.5秒** + `easeLinearity: 0.25` 最舒适——有一个加速→减速的缓动过程，视觉上像在"飞"。

**3. 三部故事互通导航**

用户看完张骞故事第5章，看到"佛教沿此路东传"的提示，点击可以直接跳转到玄奘故事的第3章。

实现很简单：URL 参数控制。`story3.html?chapter=3` 直接跳到指定章节。

三部故事共享同一份城市坐标数据和同一张底图配置，URL 互通只用了不到 20 行代码。

## 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 地图 | Leaflet.js | 轻量，免费瓦片，移动端友好 |
| 动画 | Leaflet flyTo + easeLinearity | 飞越动画只需一行 API |
| 叙事卡片 | 纯 CSS absolute + transition | 不需要任何 UI 框架 |
| 数据 | 单 JSON 文件 | 35城23人物24物产255边，43KB |
| 部署 | GitHub Pages | push 即上线，零成本 |

全程纯前端，零依赖框架。Leaflet + 原生 JS，不到 500 行代码完成一个完整叙事引擎。

## 写在最后

做这个项目的过程中，我有一个越来越强烈的感受：

**中国有太多值得讲的故事，但很多时候，故事没有被讲好。**

我们这代前端能做的不只是写 CRUD 和调样式。我们能用地图、动画、数据可视化，把那些埋没在百科词条和学术论文里的故事，变成一个任何人都能沉浸体验的东西。

玄奘走了17年。我做这个故事只花了三天。

能用代码让更多人看到这条路上发生过的奇迹，这本身就是一件很值得的事。
